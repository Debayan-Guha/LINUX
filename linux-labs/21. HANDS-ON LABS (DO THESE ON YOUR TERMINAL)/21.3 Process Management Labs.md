## 21.3.1.9 Use `ps -eo pid,ppid,cmd,%mem,%cpu --sort=-%cpu` to see CPU hogs

This command shows processes sorted by CPU usage (highest first) - perfect for finding what's consuming your CPU!

<img width="1366" height="768" alt="image" src="https://github.com/user-attachments/assets/caae6a0b-b448-4c07-8fc7-0ca138ef2ee8" />

BreakDown 
* **`ps`**: Reports a snapshot of the current processes.
* **`-e`**: Selects all processes running on the system.
* **`-o`**: Specifies a user-defined custom output format.
* **`pid,ppid,cmd,%mem,%cpu`**: Defines the columns to display:
  * `pid`: Process ID
  * `ppid`: Parent Process ID
  * `cmd`: Executable name/command
  * `%mem`: Percentage of physical memory used
  * `%cpu`: Percentage of CPU time used
* **`--sort=-%cpu`**: Sorts the results by CPU usage in descending order (highest first due to the `-` prefix).


## 21.3.1.10 Observe process state (R, S, D, Z) using `ps`
Process states show what a process is currently doing. Think of them like status indicators for each running program.

<img width="1366" height="768" alt="image" src="https://github.com/user-attachments/assets/5bb6914f-cf70-422e-8a87-22f8b08f2ade" />

**Linux Process States Explained**

When monitoring processes with tools like `ps` or `top`, the kernel reports the execution status of each process using a specific state symbol.

**Process State Table**

| State | Symbol | Meaning | Example |
| :---: | :---: | :--- | :--- |
| **R** | Running | Process is actively using CPU | `sleep 1` (briefly) |
| **S** | Sleeping | Waiting for something (I/O, event) | `sleep 300`, `ssh`, `bash` |
| **D** | Uninterruptible Sleep | Waiting for disk I/O (cannot be killed) | Reading/writing to a slow disk |
| **Z** | Zombie | Process finished, but parent has not cleaned up | Defunct processes |
| **T** | Stopped | Paused by a system signal | Job paused via `Ctrl+Z` |
| **I** | Idle | Kernel thread sleeping to save power | `kthreadd` |
| **X** | Dead | Process is dying and about to be cleaned up | Rare transient state |

**Note :-** When you run a command in the background using the `&` operator, it becomes a job managed by the shell session in which it was started. The jobs command is a shell built-in, which means it only displays background jobs that belong to the current shell instance. This shell instance is tied to a specific terminal session. Therefore, if you open multiple terminal windows or tabs, each one runs its own independent shell process, and each shell maintains its own separate job table. A job started in one terminal is completely unknown to the shell running in another terminal.

* **SIGTERM (15)** = "Please stop" → The process gets a chance to clean up (save files, close connections, remove temp files) before exiting. If it's stuck, it can ignore this. Both signals kill the process, but this one does it gracefully.
* **SIGKILL (9)** = "STOP NOW!" → The kernel forcefully rips the process out of memory immediately. The process gets zero chance to clean up. It cannot ignore this. Both signals kill the process, but this one does it forcefully.

## 21.3.2.10 Start a process with lower priority using `nice`
<img width="614" height="461" alt="image" src="https://github.com/user-attachments/assets/fd2307a4-1c1e-4ff7-b174-447dd7b2627b" />

**Process Priority and the `nice` Command**

In Linux, every running process is assigned a priority level that determines how much CPU time it gets compared to other tasks. This priority scale is controlled by a numeric value known as **Niceness (NI)**, which ranges from **-20 to 19**:
* **Lower numbers (-20 to -1)** mean higher priority. The process becomes "selfish" and grabs more CPU time.
* **Higher numbers (1 to 19)** mean lower priority. The process becomes "nice" to other programs, stepping aside and letting them use the CPU first.
* **0** is the default baseline priority for standard user applications.

By default, regular users can only increase a process's niceness (making it lower priority) to prevent their long-running background tasks from slowing down the rest of the system. Only the system administrator (`root` or `sudo`) can lower the nice value to grant a process higher execution priority.

## 21.3.3.2 Run it and use `strace -p <PID>` to see system calls

The `strace -p <PID>` command hooks into an active, running process to monitor its real-time interactions with the operating system kernel. Every time the target program requests resources—such as reading files, writing to the terminal, or allocating memory—`strace` intercepts and prints that specific system call directly to your screen. This creates a live, text-based feed that is highly effective for debugging stuck programs, auditing security behaviors, and analyzing exactly how software executes at the system level.

## 21.3.3.3 Use `lsof -p <PID>` to see all open files and network connections

In Linux, "everything is a file," meaning that running programs interact with network sockets, system libraries, logs, and directories just like standard text files. You can identify exactly which process is blocking a specific port by using the command `sudo lsof -i :80`, replacing `80` with whatever port you need to check. This instantly outputs the command name and Process ID (PID) holding that port open so you can easily terminate it. If you want to view a broad list of every single active network connection across your entire system, you can drop the port number and run `sudo lsof -i` on its own. Finally, if you want to isolate only the network connections belonging to one specific process, you can combine the process and network filters using the command `sudo lsof -a -p <PID> -i`, where the `-a` operator forces the tool to show files that meet both conditions simultaneously.

**Why It Shows http Instead of 80**

By default, `lsof` translates port numbers to service names using the `/etc/services` file.

* **Port 443** = `https` (secure web)

* **Port 25** = `smtp`

## 21.3.3.5 Set `ulimit -n 100` and test file descriptor limit

`ulimit -n` = The maximum number of files your process can have open simultaneously (like how many tabs you can have open in a browser at once). It doesn't limit how many files exist on disk. It limits how many files a single process can have open at the same time.

<img width="508" height="671" alt="image" src="https://github.com/user-attachments/assets/1634c423-f1a9-47d4-84cc-20493e45a04e" />

The `error Operation not permitted` happens because normal users are allowed to lower their limits, but they cannot raise them back up, even if the new value is lower than the system's hard limit.

**Explain**
- In Linux, every process has its own open file limit, which controls how many files, sockets, and other file descriptors it can have open simultaneously. When you open a terminal, you start a shell process (like bash), and this shell acts as the parent for any commands you run from it. When you execute a script like `network_script.py`, it becomes a child process of that shell, and at the moment of its creation, it inherits its open file limit from the parent shell. 
- This means the shell and the child script each have their own separate limit value and their own open file tables — the shell has stdin, stdout, stderr, and any files it opens, while the script has its own open files like network sockets or log files. When you run `ulimit -n 100` inside the shell, you are changing the shell's own limit, and this new value will be passed down to any new child processes that are started after this change. 
- However, it does not affect processes that are already running, such as `network_script.py`, because they already inherited the old limit (e.g., 1024) when they were first created. So the rule is simple: limits are inherited from parent to child only at the time of creation, and changing the parent's limit later does not retroactively change the limits of existing children.


## 21.3.3.6 Create a process that runs after terminal closes using `nohup`
<img width="712" height="468" alt="image" src="https://github.com/user-attachments/assets/e8d7cc21-b53b-4a2b-bde3-81339addcfcd" />

When you run a script with `nohup`, it instructs the process to completely ignore the `SIGHUP` (Hang Up) signal sent by the shell when the terminal window is closed. Instead of detaching immediately, the process stays tied to the shell until the terminal exits. Once the parent shell closes, the script becomes an orphan, and the system kernel automatically re-parents it to the root init system (`systemd` with PID 1). This keeps the process safely running continuously in the background, entirely independent of any active user terminal sessions.


<img width="1366" height="768" alt="image" src="https://github.com/user-attachments/assets/6c74e9b5-d99d-4cfd-9ebf-0adecf309498" />


## 21.3.3.7 Detach a running process using `disown`

```bash
ps -o ppid= -p <PID>
```
`disown` removes a job from the shell's job table so it won't receive `SIGHUP` when the terminal closes. The process continues running in the background, detached from the shell.
