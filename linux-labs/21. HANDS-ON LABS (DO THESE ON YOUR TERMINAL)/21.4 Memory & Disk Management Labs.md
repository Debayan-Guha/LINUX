## 21.4.1.2 Run `vmstat 2 5` and understand each column (r, b, si, so, us, sy, id)
<img width="892" height="259" alt="image" src="https://github.com/user-attachments/assets/f0d2b240-72e2-43f2-a6c3-6936bfa83bab" />

`vmstat 2 5` stands for **Virtual Memory Statistics**. It is used to monitor system performance by displaying a summary of your operating system's CPU, memory, swap, and disk activity. The numbers `2 5` tell the system to take a measurement every **2 seconds**, for a total of **5 times**, before automatically stopping.

| Category | Column | Name | Description |
| :--- | :--- | :--- | :--- |
| **Process** | **r** | Runnable | The number of processes currently running or waiting for CPU time. If this number is consistently higher than your CPU core count, your processor is bottlenecked. |
| **Process** | **b** | Blocked | The number of processes trapped in uninterruptible sleep, usually waiting for slow disk I/O (Input/Output) operations to complete. |
| **Swap** | **si** | Swapped In | The amount of memory being moved *from* the disk (Swap space) back *into* your physical RAM per second. |
| **Swap** | **so** | Swapped Out | The amount of memory being moved *out* of physical RAM and written *to* the disk per second because your RAM is full. High numbers here mean your system is running out of physical RAM. |
| **CPU** | **us** | User Time | The percentage of CPU time spent running non-kernel user applications (like your python scripts, browser, or desktop environment). |
| **CPU** | **sy** | System Time | The percentage of CPU time spent running kernel-level system tasks (like managing memory, hardware I/O, or system calls). |
| **CPU** | **id** | Idle Time | The percentage of CPU time that is completely unused and free. A high number means your processor is resting. |

## 21.4.1.8 Run `stress --vm 1 --vm-bytes 512M` to simulate memory pressure

```bash
stress --vm 1 --vm-bytes 512M
```
**This runs a stress test that:**
* **`--vm 1`** = Creates 1 process that allocates memory
* **`--vm-bytes 512M`** = Allocates 512 MB of memory per process

<img width="869" height="572" alt="image" src="https://github.com/user-attachments/assets/390d0c44-9590-448c-aeab-d3cf7361d91b" />

**Why the `stress` Command Spawns Two Process IDs**

The `stress` tool uses a parent-child architecture to isolate control signals from active heavy work:

* **Parent Process (Main Manager)**: Handles runtime tracking, checks parameters, and monitors signals.
* **Child Worker Process (The "Hog")**: Spawned via a `fork()` system call to aggressively allocate and churn the targeted memory (e.g., 512 MB).

Increasing the worker count (e.g., `--vm 4`) will generate one parent process alongside four distinct child process IDs.

## 21.4.1.10 Clear page cache: `sync && echo 3 | sudo tee /proc/sys/vm/drop_caches`
<img width="775" height="431" alt="image" src="https://github.com/user-attachments/assets/1b4b7798-b497-493c-8f2e-61dfacd875bc" />

**Memory Cache Clearing Mechanism**

The cache clearing pipeline uses sequential operations to securely flush volatile memory directly onto physical storage before forcing a kernel buffer reset.

| Part | What it does |
| :--- | :--- |
| **sync** | Flushes all pending writes to disk (syncs filesystem buffers) |
| **echo 3** | Tells kernel to drop all caches (pagecache, dentries, inodes) |
| **sudo tee** | Writes to `/proc/sys/vm/drop_caches` with root privileges |
| **/proc/sys/vm/drop_caches** | Kernel interface to clear caches |
