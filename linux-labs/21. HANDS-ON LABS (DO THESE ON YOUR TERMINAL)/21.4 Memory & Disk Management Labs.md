## 21.4.1.2 Run `vmstat 2 5` and understand each column (r, b, si, so, us, sy, id)
<img width="892" height="259" alt="image" src="https://github.com/user-attachments/assets/f0d2b240-72e2-43f2-a6c3-6936bfa83bab" />

`vmstat 2 5` stands for **Virtual Memory Statistics**. It is used to monitor system performance by displaying a summary of your operating system's CPU, memory, swap, and disk activity. The numbers `2 5` tell the system to take a measurement every **2 seconds**, for a total of **5 times**, before automatically stopping.

| Category | Column | Name | Description |
| :--- | :--- | :--- | :--- |
| **Process** | **r** | Runnable | The number of processes currently running or waiting for CPU time. If this number is consistently higher than your CPU core count, your processor is bottlenecked. |
| **Process** | **b** | Blocked | The number of processes trapped in uninterruptible sleep, usually waiting for slow disk I/O (Input/Output) operations to complete. |
| **Swap** | **si** | Swapped In | The amount of memory being moved *from* the disk (Swap space) back *into* your physical RAM per second. |
| **Swap** | **so** | Swapped Out | The amount of memory being moved *out* of physical RAM and written *to* the disk per second because your RAM is full. High numbers here mean your system is running out of physical RAM. |
| **CPU** | **us** | User Time | The percentage of CPU time spent running non-kernel user applications (like your python scripts, browser, or desktop environment). |
| **CPU** | **sy** | System Time | The percentage of CPU time spent running kernel-level system tasks (like managing memory, hardware I/O, or system calls). |
| **CPU** | **id** | Idle Time | The percentage of CPU time that is completely unused and free. A high number means your processor is resting. |

## 21.4.1.8 Run `stress --vm 1 --vm-bytes 512M` to simulate memory pressure

```bash
stress --vm 1 --vm-bytes 512M
```
**This runs a stress test that:**
* **`--vm 1`** = Creates 1 process that allocates memory
* **`--vm-bytes 512M`** = Allocates 512 MB of memory per process

<img width="869" height="572" alt="image" src="https://github.com/user-attachments/assets/390d0c44-9590-448c-aeab-d3cf7361d91b" />

**Why the `stress` Command Spawns Two Process IDs**

The `stress` tool uses a parent-child architecture to isolate control signals from active heavy work:

* **Parent Process (Main Manager)**: Handles runtime tracking, checks parameters, and monitors signals.
* **Child Worker Process (The "Hog")**: Spawned via a `fork()` system call to aggressively allocate and churn the targeted memory (e.g., 512 MB).

Increasing the worker count (e.g., `--vm 4`) will generate one parent process alongside four distinct child process IDs.

## 21.4.1.10 Clear page cache: `sync && echo 3 | sudo tee /proc/sys/vm/drop_caches`
<img width="775" height="431" alt="image" src="https://github.com/user-attachments/assets/1b4b7798-b497-493c-8f2e-61dfacd875bc" />

**Memory Cache Clearing Mechanism**

The cache clearing pipeline uses sequential operations to securely flush volatile memory directly onto physical storage before forcing a kernel buffer reset.

| Part | What it does |
| :--- | :--- |
| **sync** | Flushes all pending writes to disk (syncs filesystem buffers) |
| **echo 3** | Tells kernel to drop all caches (pagecache, dentries, inodes) |
| **sudo tee** | Writes to `/proc/sys/vm/drop_caches` with root privileges |
| **/proc/sys/vm/drop_caches** | Kernel interface to clear caches |

## 21.4.2.2 Run `df -i` to check inode usage
<img width="621" height="257" alt="image" src="https://github.com/user-attachments/assets/5ed690b5-93f0-42af-85c0-42331a082ca7" />

**1: What are inodes?**

An **inode** (index node) is a data structure on a Linux filesystem that stores all the metadata about a file or directory except its actual name and data contents. It tracks attributes like file size, owner permissions, timestamps, and the exact physical disk block locations where the file's data is stored. Every single file and directory requires exactly one inode to exist.

**BreakDown**
* **1 File** = Consumes **1 Inode** (stores permissions, size, and data locations)
* **1 Folder** = Consumes **1 Inode** (stores permissions, folder size, and the list of contents inside it)

**2: What's the difference between disk space and inodes?**

* **Disk space** = Data storage capacity (the raw size of file contents, measured in MB/GB/TB)
* **Inodes** = Metadata storage capacity (the total maximum number of files/directories allowed on the drive)

*Note: You can completely run out of inodes even if you still have gigabytes of free disk space remaining!*

**3: Why would a system run out of inodes?**

This is typically caused by having too many small files on the drive, which consumes all available metadata slots. Common culprits include:
* Excessive or unrotated log files
* Large email spool directories
* Bloated application caches
* Thousands of tiny web cache files

**4: How to fix high inode usage?**

Delete unnecessary small files to free up metadata slots, focusing heavily on:
* Clearing out old logs and temporary directory files
* Purging application and web caches
* Finding and removing massive nested directories containing millions of empty files

**Navigating the Interface**

After the scan finishes, you'll see an interactive list of directories and files sorted by size, from largest to smallest. You can navigate through this list using the following keys:

* **↓ / ↑ (Arrow keys) or `j` / `k`**: Move the cursor up and down through the items.
* **Enter or → (Right arrow)**: Open the selected directory.
* **← (Left arrow) or `h`**: Go back to the parent directory.
* **`q`**: Quit the `ncdu` interface.

**Interactive Features**

`ncdu` also provides shortcuts for sorting, viewing details, and deleting files directly from the interface.

* **Sorting by Size**: Press `s` to sort items by file size (press again to reverse the order).
* **Sorting by Name**: Press `n` to sort items alphabetically by name.
* **File Information**: Press `i` to see detailed information like the path and disk usage of a selected file or directory.
* **Deleting Items**: Select an item and press `d`. You will be asked to confirm the deletion before it's permanently removed.

## 21.4.2.7 List block devices: `lsblk`
<img width="366" height="658" alt="image" src="https://github.com/user-attachments/assets/84d81196-f0b0-4024-ac78-283fad9bb669" />

`lsblk` (List Block Devices) shows all storage devices (disks, partitions, and their mount points) in a tree format.

| Column | Meaning | Your Value |
| :--- | :--- | :--- |
| **NAME** | Device name | `sda`, `sda1`, `sda2` |
| **MAJ:MIN** | Major:Minor device numbers | `8:0`, `8:1`, `8:2` |
| **RM** | Removable (1=yes, 0=no) | `0` (not removable) |
| **SIZE** | Size of device | `238.5G`, `1G`, `237.4G` |
| **RO** | Read-only (1=yes, 0=no) | `0` (read-write) |
| **TYPE** | Device type | `disk`, `part`, `loop` |
| **MOUNTPOINTS** | Where it's mounted | `/`, `/boot/efi` |

| Type | Meaning | Example |
| :--- | :--- | :--- |
| **disk** | Physical disk | `sda` — your SSD/HDD |
| **part** | Partition on a disk | `sda1`, `sda2` |
| **loop** | Loop device (snap packages) | `loop0`, `loop1`... |


**Loop Devices**
```text
loop0   → /snap/bare/5
loop1   → /snap/brave/656
loop2   → /snap/core/17284
...
```

Loop devices are virtual drives created by Snap packages. Each snap application is isolated inside its own standalone package filesystem and mounted dynamically onto your operating system tree as a read-only loop device.


## 21.4.2.8 Mount a new temporary filesystem: `sudo mount -t tmpfs tmpfs /mnt/tmp`

* **`-t tmpfs`**: Specifies the filesystem type. `tmpfs` stands for "Temporary File System," which is Linux's built-in RAM disk driver.
* **`tmpfs`**: The source device name. Since there is no physical hard drive partition (like `/dev/sda1`), we just pass the name `tmpfs` as a virtual device placeholder.
* **`/mnt/tmp`**: The target folder (mount point) on your computer where this RAM disk will be attached. You must create this folder using `sudo mkdir -p /mnt/tmp` before running the mount command.

<img width="577" height="173" alt="image" src="https://github.com/user-attachments/assets/f1e6ab4f-efeb-4f25-b33c-b302f8c13056" />
<img width="577" height="173" alt="image" src="https://github.com/user-attachments/assets/ce20c31c-2423-4769-9644-39a239a602c1" />

**`tmpfs` (Temporary Filesystem)**

**1: What is tmpfs?**

A temporary filesystem that resides in RAM. It's fast but volatile — data is lost on reboot.

**2: When would you use tmpfs?**

For temporary files, caches, or high-speed I/O where persistence isn't needed.

**3: How to limit tmpfs size?**

mount -t tmpfs -o size=500M tmpfs /mnt/tmp`


## 21.4.2.9 Check `/etc/fstab` to understand persistent mounts
<img width="670" height="309" alt="image" src="https://github.com/user-attachments/assets/bbe24d40-cefe-4fea-a3ec-45d74c34f53c" />

**`/etc/fstab` (Filesystem Table)**

**1: What is /etc/fstab?**

The filesystem table — a system configuration file that defines how and where storage devices are mounted automatically at boot.

**2: What are the columns in fstab?**

Device, mount point, filesystem type, options, dump, pass.

**3: What's the difference between pass values 1 and 2?**

1 = root filesystem (checked first), 2 = other filesystems (checked after root).
