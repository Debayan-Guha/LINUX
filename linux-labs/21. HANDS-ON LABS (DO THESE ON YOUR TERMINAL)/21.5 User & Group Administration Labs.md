## 21.5.1.1 Create a user `devops_user` with home directory `/home/devops_user`
<img width="680" height="179" alt="image" src="https://github.com/user-attachments/assets/131df6fb-b02b-4a8a-8447-253667643355" />

**Users are system-wide.**  

Once you create a user (e.g., `devops_user`), that user exists everywhere on the system — you can log in from any terminal, any directory, or any session.

**What the `-d /home/devops_user` Option Actually Does**

It does **NOT** create a user inside `/home/`. It simply sets the target path layout location for that specific user’s home directory profile.

| Part | Meaning |
| :--- | :--- |
| **`sudo`** | Runs the command with root privileges (required for system-wide account modifications). |
| **`useradd`** | The base system utility used to initialize and register a new user account. |
| **`-m`** | Tells the system: “Create that physical folder structure on the disk right now.” |
| **`-d /home/devops_user`** | Tells the system: “When this user logs in, their home folder will be `/home/devops_user`.” |
| **`devops_user`** | Specifies the unique username for the new system-wide account. |
Use code with caution.

**Verification Commands for New Linux Users**

| Command | Purpose | Verification target |
| :--- | :--- | :--- |
| `id <username>` | Confirms account existence | Displays active UID, GID, and assigned security groups. |
| `getent passwd <username>` | Checks system databases | Displays registration entry details, home folder path, and login shell. |
| `ls -ld /home/<username>` | Validates space isolation | Confirms the physical home folder directory layout exists on disk. |



**Creating a user with a dedicated home directory achieves full isolation across two distinct boundaries:**
- **Identity Isolation (Account)**: The system treats the user as an independent security entity with restricted permissions, preventing them from interacting with system files or other users' processes by default.
- **Space Isolation (Sandbox)**: The home folder acts as a private storage sandbox. It isolates configuration files, bash environments, and data caches, securing them from unauthorized access by other non-privileged users.
Use code with caution.

| Command | Purpose |
| :--- | :--- |
| **`sudo passwd devops_user`** | Set/change password |
| **`sudo passwd -S devops_user`** | Check password status |
| **`sudo passwd -e devops_user`** | Force password change on next login |
| **`sudo passwd -l devops_user`** | Lock user account |
| **`sudo passwd -u devops_user`** | Unlock user account |
| **`sudo passwd -d devops_user`** | Remove password |

**Account Locking vs. Unlocking**

Locking an account is a non-destructive way to revoke system access without purging data.
* **Locking (`passwd -l`)**: Prepends an exclamation mark (`!`) to the user's encrypted password string in `/etc/shadow`. This breaks password matching and blocks interactive logins, but leaves all files, home directories, and data intact.
* **Unlocking (`passwd -u`)**: Removes the restriction character from the password string, restoring the original credential and instantly re-enabling system access.



**`sudo passwd -S` Output Field Breakdown**

The remaining numbers (`0 99999 7 -1`) control the system's password expiration and security aging policies. 

| Position | Value | Name | Description |
| :--- | :--- | :--- | :--- |
| **1** | `devops_user` | Username | The name of the account being checked. |
| **2** | `P` | Status Flag | Account status (`P` = Password set, `L` = Locked, `NP` = No password). |
| **3** | `07/22/2026` | Change Date | The exact date the password was last modified. |
| **4** | `0` | Minimum Age | Minimum days required between password changes (`0` = can change anytime). |
| **5** | `99999` | Maximum Age | Maximum days the password is valid (`99999` = effectively never expires). |
| **6** | `7` | Warning Period | Number of days before expiration that the user receives warning alerts. |
| **7** | `-1` | Inactivity Limit | Days after expiration before the account freezes completely (`-1` = disabled/never freezes). |



## 21.5.1.3 Create a user `app_user` with no login shell (`/usr/sbin/nologin`)
<img width="703" height="150" alt="image" src="https://github.com/user-attachments/assets/be0089d4-e147-4d9b-9784-2cad9eb66c8e" />

* **`-r`**: Spawns a **System Account** with a low UID (typically < 1000) so it stays hidden from graphical login screens.
* **`-M`**: Skips creating a traditional `/home/` directory, as background services do not require personal user profiles.
* **`-s /usr/sbin/nologin`**: Sets the default shell to lock out interactive console execution. *(Note: `/bin/false` can also be used as a silent alternative).*


**1. The Core Concept of a Non-Login Shell**

A user account configured with a non-login shell cannot be accessed interactively via a terminal or SSH. This setup isolates background system processes without providing human access to the operating system tools.
* **What `/usr/sbin/nologin` does**: It actively prevents interactive console or remote SSH logins. If an unauthorized actor tries to log into that user profile, the terminal drops the connection.
* **What it does NOT do**: It does **not** stop the system from executing applications, handling background tasks, or reading configuration assets as that user.
* **How processes are run**: The system root kernel or initialization managers (`systemd`) can safely initiate system processes under this user profile. You can also execute targeted ad-hoc system commands using administrative escalations:
  ```bash
  sudo -u service_user <command>
  ```

**2: Why use -M with -r?**

System users typically don't need home directories — -M prevents creating one.

**3. Why Use Service Users? (Damage Control Analysis)**

While any daemon or process can technically run with unrestricted `root` system authority, dropping process privileges down to a standalone user account isolates files and keeps breaches contained.

| If the background service runs as... | And it gets maliciously compromised, the attacker can... |
| :--- | :--- |
| **`root`** | Wipe out the entire filesystem, deploy persistent rootkits, access all raw memory segments, and modify kernel behaviors. |
| **Your personal user account** | Delete or steal all your home directory data, harvest your private SSH deployment keys, inspect local email files, and compromise browser credentials. |
| **A dedicated `nologin` service user** | **Only access the isolated files and resource buckets explicitly owned by that specific service.** The rest of the system files remain protected. |


**4. Key Benefits of a Service User**

| Benefit Element | Operational Explanation |
| :--- | :--- |
| **Security (Least Privilege)** | The process runs with only the bare-minimum permissions needed to handle its data—nothing more. This inherently limits the blast radius of a breach. |
| **No Shell Access** | Configuring the profile to `/usr/sbin/nologin` closes interactive console routes. Attackers cannot execute commands to browse files. |
| **Clear File Ownership** | All related logs, databases, and configuration settings are owned by that specific ID, simplifying backup tasks and access audits. |
| **Process Isolation** | System metrics tracking tools (like `ps aux` or `htop`) clearly tag which background jobs belong to which app infrastructure context. |
| **Easy Cleanup** | Completely purging an application setup becomes a single clean execution step using `userdel -r`. |
| **Resource Limits** | System constraints (`ulimit` configurations or kernel `cgroups`) can be applied uniquely to the service profile to prevent memory or CPU leaks from freezing the whole server. |
| **Audit and Compliance** | Security tracking facilities map file operations and networking events to unique UIDs, which is a core requirement for technical audits. |


**5. If i run an app from a folder that only the root user can access, is that just as secure as using a dedicated system user with no login?**

No, running an app as root inside a restricted folder is not as secure, because if the app is compromised, the attacker gains root privileges and full control over the system.

## 21.5.1.5 Modify `devops_user` to have expiry date 30 days from now

```bash
// Set No Expiry (Remove Expiry):
sudo chage -E -1 devops_user

// Set Immediate Expiry (Lock Account):
sudo chage -E 0 devops_user

// Force Password Change on Next Login:  This sets the last password change to 0, forcing the user to change password on next login.
sudo chage -d 0 devops_user
```

<img width="729" height="388" alt="image" src="https://github.com/user-attachments/assets/d1fbf08f-26db-4fc4-ad19-ec855b27ea36" />

**Epoch Time Interpretation Error & Resolution**

Running the command with a raw integer causes `chage` to parse the number relative to the Unix Epoch rather than calculating a dynamic offset from the current calendar day.

| Aspect | Details |
| :--- | :--- |
| **The Error** | `sudo chage -E 30 devops_user` |
| **Why It Shows 1970** | The system interprets `30` as **30 days since January 1, 1970** (Unix Epoch), setting the expiration to January 31, 1970. Because this date is in the past, the user is locked out instantly. |
| **The Correct Approach** | `sudo chage -E $(date -d "+30 days" +%Y-%m-%d) devops_user` |
| **How Resolution Works** | The `$(date -d "+30 days" +%Y-%m-%d)` sub-command dynamically calculates the date exactly 30 days from today and formats it as `YYYY-MM-DD` for `chage` to read accurately. |

<img width="729" height="388" alt="image" src="https://github.com/user-attachments/assets/e04a1ccf-000f-4626-a1de-0a191c8f0d8c" />



**Linux Account Expiration and Management policies**

Setting an account expiration date defines a strict timeline after which system access is automatically revoked by the authentication system.

**Purpose and Operational Logic**
* **Mechanism**: The user `devops_user` will be locked out automatically on `2026-08-21`. They cannot log in after that date.
* **Why Set Expiry?**: It reduces security liabilities by guaranteeing that temporary accesses are automatically cleaned up without requiring manual administrator intervention.

**Business Drivers for Account Expiration**
| Reason | Operational Example |
| :--- | :--- |
| **Temporary Employees** | Interns or contractors whose account should expire exactly when their contract ends. |
| **Project-Based Access** | Restricting resource access strictly to the active lifecycle window of a project. |
| **Security Compliance** | Ensuring stale or forgotten accounts do not remain active indefinitely on production nodes. |
| **Audit Requirement** | Meeting strict regulatory compliance standards that demand systematic account expiration. |
| **Seasonal Access** | Provisioning access for seasonal workers that cleanly self-terminates at the end of the season. |


**What Happens After Expiry?**

When a user attempts to authenticate after the set expiration threshold, the system boundary drops the session immediately.

**System Restrictions Imposed Upon Expiration:**
* ❌ Login via SSH is completely blocked
* ❌ Login via terminal console is rejected
* ❌ Administrative privilege switching via `su` or `sudo` is denied

| Expiry Type | System Behavior | Implementation Command |
| :--- | :--- | :--- |
| **Password Expiry** | Forces the user to securely change their password during their next interactive login session. | `sudo chage -d 0 devops_user` |
| **Account Expiry** | Completely locks out the account, denying any forms of system authentication after the specified date. | `sudo chage -E 2026-08-21 devops_user` |




## 21.5.1.6 Lock the `app_user` account using `usermod -L`
<img width="665" height="254" alt="image" src="https://github.com/user-attachments/assets/56d4cfc8-0650-4fbf-bfaf-4f54b8b083fc" />

A user without a password is automatically locked. The `!` in `/etc/shadow` means locked — whether from `usermod -L` or because no password was ever set.

To unlock any account, a password must exist first. Without a password hash, the system cannot unlock the account because unlocking would create a passwordless login, which is a security risk.

For service accounts (with `/usr/sbin/nologin`), a password is not required — they are locked by default and should stay locked. They don't need to be unlocked because they are not meant for interactive login.


Both `passwd -l` and `usermod -L` achieve the exact same operational result — they lock a user account by prepending an exclamation mark (`!`) to the encrypted password hash string inside the `/etc/shadow` security file.

| Command | Operational Effect | Resulting Entry Structure in `/etc/shadow` |
| :--- | :--- | :--- |
| **`sudo passwd -l app_user`** | Locks the user account | `app_user:!$6$xyz...` |
| **`sudo usermod -L app_user`** | Locks the user account | `app_user:!$6$xyz...` |
| **`sudo passwd -u app_user`** | Unlocks the user account | `app_user:$6$xyz...` |
| **`sudo usermod -U app_user`** | Unlocks the user account | `app_user:$6$xyz...` |

* **`passwd`** focuses strictly on password-related operations and credential boundaries.
* **`usermod`** focuses on holistic user attribute modifications, providing locking flags so system administrators can modify and lock a profile simultaneously without switching tools.

## 21.5.3.1 Grant `devops_user` permission to run `systemctl` as root

```bash
sudo visudo -f /etc/sudoers.d/devops_user

// add this line
devops_user ALL=(ALL) NOPASSWD: /usr/bin/systemctl

// remove permission then remove that file
sudo rm /etc/sudoers.d/devops_user

// check and validate the syntax and permissions of your sudoers files without actually opening or editing them
sudo visudo -c
```

- `devops_user ALL=(ALL) NOPASSWD: /usr/bin/systemctl`
  
| Part | Meaning |
| :--- | :--- |
| **`devops_user`** | The specific user account that is being granted administrative privileges. |
| **`ALL`** | The host machine specification. This means the user can execute the command from **any host** or terminal network. |
| **`(ALL)`** | The target user execution context. This allows the user to run the command **as any user** (including root). |
| **`NOPASSWD:`** | Security bypass flag. The user is **not required to enter their password** when executing this specific command. |
| **`/usr/bin/systemctl`** | Command restriction. The user is strictly limited to running **only this specific binary command** (used for managing system services). |

- `sudo visudo -f /etc/sudoers.d/devops_user`
  
| Part | Meaning |
| :--- | :--- |
| **`sudo`** | Executes the entire command with administrative (root) privileges, which is required to modify system security configurations. |
| **`visudo`** | The safe utility program used to edit sudoers files. It locks the file from simultaneous edits and **checks for syntax errors** before saving to prevent lockouts. |
| **`-f`** | The "file" option flag that instructs `visudo` to bypass the default main file and edit a specific target path instead. |
| **`/etc/sudoers.d/devops_user`** | The absolute file path to the custom configuration. Keeping rules in separate files inside `/etc/sudoers.d/` keeps system configuration modular and clean. |

**1. Why creating sudoers.d file?**

Creating a sudoers file (or adding entries to it) allows you to grant specific users or groups administrative privileges without sharing the root password.

**2. Why `visudo -c` Flags File Permissions?**

Linux requires sudoers configurations to be incredibly strict. Even though "others" cannot read your file, the permission state `-rw-r-----` evaluates to `0640` because of the `w` (write) permission assigned to the root user.

**The Hard Security Rule**

Sudo enforces a strict security policy: **No file inside the `/etc/sudoers.d/` directory is allowed to be writable by anyone at all—not even by the root user.**

**Why This Rule Exists**

* **Safety and Integrity:** The file must be strictly read-only to prevent accidental edits, background scripts, or rogue applications from modifying administrative rights.
* **Enforced Verification:** Forcing the file to be read-only ensures that changes can only be made using the safe `visudo` command, which validates your syntax before applying updates to the system.

## 21.5.3.7 Set sudo timeout to 15 minutes

This configuration sets a 15-minute timing window for that particular user, which triggers only when they run a sudo command. The countdown timing starts immediately after that first successful sudo command and password entry. If the user remains active and keeps running more sudo commands, the 15-minute timer automatically resets back to the beginning each time, allowing them to work without typing the password again. However, if the user stops using sudo and 15 continuous minutes of inactivity pass, the timer will hit zero and clear the password memory. If they do not use sudo at all, the timing will not count, and nothing happens. The next time they try to use sudo after the timer runs out, the terminal will simply prompt them to enter their password again. In other words, if the user uses sudo for the first time and does not use sudo again, then after 15 minutes the system will ask for the password. If the user uses sudo again after the first time, then the timer will reset to 15 minutes for each subsequent sudo command.

```bash
// for specific user
Defaults:devops_user timestamp_timeout=15

// for global
Defaults timestamp_timeout=15
```
