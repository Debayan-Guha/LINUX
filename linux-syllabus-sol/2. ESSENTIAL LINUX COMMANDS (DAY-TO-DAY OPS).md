# 2.1 File & Directory Operations

## 2.1.1 `ls` (list files with options: -l, -a, -h, -R)
- `ls -l` = permissions, ownership, size, date (auditing/filesystem checks)
- `ls -a` = includes hidden config files (.bashrc, .ssh, .env) — critical for debugging
- `ls -ltr` = sort by time, oldest last (see latest log files at bottom)



## 2.1.2 `cd` (change directory, absolute vs relative paths)

- Absolute path = starts from root `(/)` — e.g., `cd /etc/nginx` (works from anywhere).
- Relative path = starts from current location — e.g., `cd logs` (goes into logs inside where you are now).

```bash
# Go to your personal home directory (/home/debayan)
cd
# OR
cd ~

# Go directly to a specific user's home directory (replace "username" with a real name)
cd ~username

# Toggle back and forth to your previously visited directory
cd -

# Go up one level into the parent directory
cd ..

# Go up two levels into the grandparent directory
cd ../..

# Go straight to the absolute root directory of the system
cd /
```

## 2.1.4 `mkdir` (create directories, -p for parents)
```bash
# Create a single folder named "projects"
mkdir projects

# Create three separate project folders at the exact same time
mkdir {project1,project2,project3}

# Try to create a nested folder WITHOUT -p (This will FAIL and give an error)
mkdir photos/vacation/2026

# Create the entire nested folder path successfully using -p 
# (It creates "photos", then "vacation" inside it, then "2026" inside that)
mkdir -p photos/vacation/2026

# Create a master "images" folder containing three subfolders at once
mkdir -p images/{photos,videos,screenshots}
```


## 2.1.5 `rmdir` vs `rm -rf` (remove directories)

```bash
# Delete an empty folder named "old_projects"
rmdir old_projects

# Try to use rmdir on a folder with files inside (This will FAIL with an error)
rmdir photos

# Delete the "photos" folder and all its contents instantly (Use with caution!)
rm -rf photos

# DANGER: Never run this as root! It deletes your entire operating system
# sudo rm -rf /
```


## 2.1.9 `rm` (remove files, -i, -f, -r flags)
```bash
# Safely delete a file by checking with the system first
rm -i backup.txt
# Terminal will ask: remove 'backup.txt'? 

# Force delete a protected, read-only file without getting a warning prompt
rm -f old_config.conf

# Delete an entire folder named "trash_folder" along with all its contents
rm -r trash_folder

# Combine recursive and force to delete a directory instantly with zero prompts
rm -rf sample_project
```

## 2.1.10 `ln` (hard links vs soft/symbolic links)

- **hard link -** Creates a Hard Link. It creates a new name that points directly to the exact same data on the storage drive. If you delete the original file, the hard link still works and keeps the data. You cannot make hard links for folders or across different hard drives.
- **sof link -** Creates a Soft Link (also called a Symbolic Link or Symlink). It is a standard desktop shortcut that points to the original file name. If you delete the original file, the soft link breaks and stops working. You can make soft links for both files and folders.

```bash
# Create a hard link to a file named "data.txt"
ln data.txt data_hard.txt

# Create a soft link (shortcut) to the same file using the -s flag
ln -s data.txt data_soft.txt

# Create a soft link shortcut to an entire folder
ln -s /var/log/nginx/ website_logs

# View your links in the terminal (Soft links will show an arrow pointing to the source)
ls -l
# Example output line for a soft link:
# lrwxrwxrwx 1 debayan debayan 8 Jun 25 23:34 data_soft.txt -> data.txt
```

**What problem do file links solve in Linux, and how do they manage shared file access compared to copying files?**

When multiple services or processes need to access the same file simultaneously, instead of duplicating the file everywhere, we create a single reference called a link. If we copy the file to multiple locations, each copy becomes an independent file. Editing one copy does NOT affect the others — they stay outdated because they are separate files, not connected. Links solve this problem because they do NOT duplicate the file's data. Instead, a link is just a pointer that tells the system, 'Go to the original file.' Since all links point to the same original file, any edit made to the original — OR through any link — changes the source file itself. Because all links reference this single source, the change is reflected simultaneously and instantly across all services accessing it.

***Note: If processes do NOT need to access the same file simultaneously, then a single file can be used directly without creating links — there is no risk of inconsistency since only one process uses it at a time.***

Why multiple processes can't always access the same file directly:

- If two processes write to the same file at the same time, one process may overwrite the other's data.
- If one process reads while another writes, the reader may get incomplete or inconsistent data.
- File locking mechanisms can help, but they slow down performance and are complex to manage.
- That's why we need either links or proper synchronization mechanisms to manage shared file access."

## 2.2.3 `head` and `tail` (top/bottom of files)

```bash
# View the first 10 lines of the user accounts file
head /etc/passwd

# View exactly the first 5 lines of the user accounts file
head -n 5 /etc/passwd

# View the last 10 lines of a network services file
tail /etc/services

# View exactly the last 20 lines of a network services file
tail -20 /etc/services

# Monitor a live text log file in real-time (Press Ctrl+C to stop following)
tail -f /var/log/nginx/access.log
```




## 2.2.4 `tail -f`(Follow File) (real-time log following)

Monitors a text file in real-time. Instead of displaying the lines and exiting, it keeps the file open on your screen. Whenever an app or system service appends a new line to the bottom of that file, the terminal updates instantly to show it. It is the primary tool used for troubleshooting active services and tracking live errors.

```bash
# Monitor system authentication logs live (shows login attempts and sudo usage)
sudo tail -f /var/log/auth.log

# Monitor a live log file starting with the last 50 lines instead of the default 10
tail -n 50 -f /var/log/syslog

# Alternative option: tailf (does the exact same thing but saves laptop battery 
# because it only checks the file when it actually changes on disk)
tailf /var/log/nginx/error.log
```


## 2.2.5 `nl` (number lines in files)

Adds line numbers to a text file when displaying it in the terminal. It works similarly to cat -n but gives you much finer control over how blank lines and formatting are handled.

```bash
# Numbers only lines with text on them, leaving blank lines unnumbered. (This is the default setting).
nl -b t document.txt

# This does the exact same thing because "-b t" is the built-in default
nl document.txt

# Numbers every single line, including completely blank or empty lines.
nl -b a document.txt
```

## 2.2.6 `tac` (reverse order output)

**Log Analysis:** Since log files append new entries to the bottom, tac lets you view the most recent events first without scrolling.

```bash
# Display the entire file from the last line up to the first line
tac data.txt

# Reverse a system log file and pipe it into less to read recent events first
tac /var/log/auth.log | less

# Combine tac with nl to number the reversed output
tac data.txt | nl
```

## 2.2.7 `wc` (count lines, words, characters)
```bash
# Display lines, words, and bytes for a file all at once
wc notes.txt

# Count exactly how many lines are in the user accounts file
wc -l /etc/passwd

# Count the total number of words in a document
wc -w essay.txt

# Combine with other commands to count things (e.g., list files and count how many exist)
ls | wc -l
```


## 2.2.8 `file` (determine file type)

- **Security:** Detects if a dangerous script or program is disguised with a safe extension (e.g., a malware script named photo.jpg).
- **Unknown Files:** Identifies broken or extensionless files so you know which app to use to open them.

```bash
# Check the true type of a file named "document" (even if it has no extension)
file document

# Inspect a file and see clean output without the filename prefix
file -b image.png

# Output the official MIME type format (very useful for shell scripting)
file -i script.sh
# Example output: text/x-shellscript; charset=utf-8

# Test if an extension is lying (e.g., checking a script renamed to look like an image)
file fake_photo.jpg
# Example output: fake_photo.jpg: POSIX shell script text executable
```

# 2.3 File Permissions & Ownership


## 2.3.1 Understanding rwx permissions (read, write, execute)

**📊 Permission Values (The Numbers)**

| Symbol | Value | Meaning |
|--------|-------|---------|
| **r** | **4** | Read - View file contents |
| **w** | **2** | Write - Modify file contents |
| **x** | **1** | Execute - Run file as program |
| **-** | **0** | No permission |


## 2.3.2 User (u), Group (g), Others (o) concept

**👥 The Three Permission Groups**

| Group | Who It Represents |
|-------|-------------------|
| **User (u)** | The file owner (creator) |
| **Group (g)** | A collection of multiple user accounts grouped together as a team (e.g., a group named developers or finance). Every member of this group shares the exact same access rules to that file. |
| **Others (o)** | Everyone else (not owner or group) |

<img width="749" height="161" alt="image" src="https://github.com/user-attachments/assets/dfde4b83-0a1b-429e-8a62-3566eaa54585" />

**Decoding `-rw-rw-r-- 1 debayan debayan 26 Jun 23 18:55 project/src/secret.sh`**

| # | Field | Value | Meaning |
|---|-------|-------|---------|
| 1 | File Type | `-` | Regular file (not a directory) |
| 2 | User Permissions | `rw-` | Owner can **read** and **write**, not execute |
| 3 | Group Permissions | `rw-` | Group members can **read** and **write**, not execute |
| 4 | Others Permissions | `r--` | Others can **read only** |
| 5 | Hard Links | `1` | Only one link to this file|
| 6 | Owner | `debayan` | User who owns the file |
| 7 | Group | `debayan` | Group that has group permissions |
| 8 | Size | `26` | File size in bytes |
| 9 | Date | `Jun 23` | Last modified date |
| 10 | Time | `18:55` | Last modified time (6:55 PM) |
| 11 | Filename | `secret.sh` | Name of the file |




## 2.3.3 chmod with symbolic (u+x) and octal (755) modes

**Symbolic mode (letters):**

- `u` = user/owner, `g` = group, `o` = others, `a` = all
- `+` = add, `-` = `remove`, `=` = set exactly
- `r` = read, `w` = write, `x` = execute

***Examples:***

- `chmod u+x file.sh` = add execute for owner only
- `chmod go-w file` = remove write from group and others
- `chmod a=rw file` = set read+write for everyone

**Octal mode (numbers):**

- `4` = read, `2` = write, `1` = execute
- Add them: `7` = rwx (4+2+1), `6` = rw- (4+2), `5` = r-x (4+1)
- Order: owner | group | others


***Examples:***

- `chmod 755 file` = owner: rwx (7), group: r-x (5), others: r-x (5) — standard for scripts
- `chmod 644 file` = owner: rw-, group: r--, others: r-- — standard for config files
- `chmod 600 file` = owner: rw-, others: none — for sensitive files (like SSH keys)
- `chmod 700 dir` = only owner can access — for private directories

***Real-world admin rule:***

- `755` = directories and scripts (executable)
- `644` = config files (non-executable)
- `600` = private keys, credentials
- `1777` = /tmp (sticky bit) — only owner can delete their files




## 2.3.4 `chown` (change file owner and group)

**chown (Change Owner)**

Changes the user and/or group ownership of a file or directory. Because changing ownership bypasses security bounds, you must almost always use `sudo` with this command.

***Command Syntax***

| Command | Description |
|---------|-------------|
| `sudo chown username file.txt` | Changes only the **User (Owner)** of the file to the specified username. |
| `sudo chown :groupname file.txt` | Changes only the **Group** ownership of the file (note the colon `:` prefix). |
| `sudo chown username:groupname file.txt` | Changes **both** the User and the Group at the exact same time in a single command. |
| `sudo chown -R username:groupname folder_name` | Changes ownership **recursively** (`-R` stands for recursive). Applies the change to the folder and **every** file and subfolder inside it. |

***Examples in Terminal***

```bash
# Change the owner of a file named "data.txt" to the user "debayan"
sudo chown debayan data.txt

# Change only the group ownership of a file to the "developers" team
sudo chown :developers data.txt

# Change both the owner to "debayan" and the group to "developers" simultaneously
sudo chown debayan:developers report.txt

# Recursively change ownership for an entire web project folder and all its contents
sudo chown -R www-data:www-data /var/www/html/
```


## 2.3.5 chgrp (change group ownership)

**chgrp (Change Group)**

Changes only the **Group** ownership of a file or directory. While `chown` can also change groups, `chgrp` is a dedicated tool specifically for managing team access without altering the file's primary owner.

***Key Rules***

| Rule | Description |
|------|-------------|
| **Privileges** | Standard users can use `chgrp` to change a file's group, **but only if** they own the file and are already a member of the target group. |
| **Admin Access** | If you need to assign a file to a group you do **not** belong to, you must prefix the command with `sudo`. |

***Command Syntax***

| Command | Description |
|---------|-------------|
| `chgrp groupname file.txt` | Changes group ownership of a single file. |
| `sudo chgrp groupname file.txt` | Forces group change even if you're not a member of the target group. |
| `chgrp -R groupname folder/` | Changes group ownership **recursively** (`-R`) for the entire directory and all contents. |

***Examples in Terminal***

```bash
# Change the group of "notes.txt" to the "developers" group
chgrp developers notes.txt

# Use sudo to force change a system file group to "admin"
sudo chgrp admin /var/www/config.php

# Change the group recursively for an entire directory and all files inside (-R)
chgrp -R testing_team automation_project/
```


## 2.3.6 Special permissions: `SUID (4)`, `SGID (2)`, `Sticky Bit (1)`


**Special Permissions: SUID, SGID, and Sticky Bit**

Linux uses three special permission flags alongside standard `rwx` rules. They handle specific security and administrative tasks. You can represent them using **symbols** or a **fourth number** placed at the very front of a standard three-digit `chmod` value.


- **SUID (Set Owner User ID) — Value: 4**

When you set SUID on an **executable file**, any regular user running that file temporarily gains the privileges of the **file owner** (usually root), instead of running it with their own permissions.

**Symbolic Representation:** An `s` appears in the owner's execute field (e.g., `-rwsr-xr-x`).

**Why it is needed:** Regular users need to change their passwords, which requires editing the protected `/etc/shadow` file. The `/usr/bin/passwd` command has SUID set so it can write to that system file safely on your behalf.


- **SGID (Set Group ID) — Value: 2**

SGID works on both files and folders, but its primary administrative use is on **folders (directories)**. When you set SGID on a folder, any new file or subfolder created inside it automatically **inherits the group owner** of the parent folder, regardless of who created it.

**Symbolic Representation:** An `s` appears in the group's execute field (e.g., `drwxrwsr-x`).

**Why it is needed:** Essential for shared team directories. It ensures that any file created by User A can instantly be read or modified by User B if they belong to the same group.


- **Sticky Bit — Value: 1**

The Sticky Bit is applied strictly to **shared folders**. When you set the Sticky Bit on a directory, **only the owner of a file** (or root) can delete or rename that file inside the directory. Other users cannot touch it, even if they have full write access to the folder itself.

**Symbolic Representation:** A `t` appears in the others' execute field (e.g., `drwxrwxrwt`).

**Why it is needed:** Secures public storage locations. This is exactly how the `/tmp` folder stays safe; everyone can create files there, but nobody can maliciously delete someone else's work.


***Quick Reference Table***

| Special Permission | Numeric Value | Symbol | Effect | Common Use |
|--------------------|---------------|--------|--------|------------|
| **SUID** | 4 | `s` on owner's `x` | Run file as **owner** | `/usr/bin/passwd` |
| **SGID** | 2 | `s` on group's `x` | Inherit **parent group** | Shared team directories |
| **Sticky Bit** | 1 | `t` on others' `x` | Only **owner** can delete | `/tmp` directory |


***Examples in Terminal***

```bash
# Set SUID on a custom script (using the numeric 4 at the front)
sudo chmod 4755 custom_admin_tool.sh
# Symbol alternative: chmod u+s custom_admin_tool.sh

# Set SGID on a collaborative team folder (using the numeric 2)
sudo chmod 2770 /home/shared_project
# Symbol alternative: chmod g+s /home/shared_project

# Set the Sticky Bit on a public drop-box folder (using the numeric 1)
chmod 1777 /home/public_dropbox
# Symbol alternative: chmod o+t /home/public_dropbox

# View how these special permissions look in long format output
ls -l
# Example outputs:
# -rwsr-xr-x 1 root root 42560 Jun 26 11:45 custom_admin_tool.sh  (SUID 's')
# drwxrws--- 2 root developers 4096 Jun 26 11:45 shared_project    (SGID 's')
# drwxrwxrwt 2 root root       4096 Jun 26 11:45 public_dropbox    (Sticky 't')
```


## 2.3.7 Understanding `umask` (default permission mask)

**Umask** is a automatic, system-wide default permission filter. It subtracts specific permissions the exact moment a new file or folder is created so they are not born completely exposed to other users.

**Common umask Values**

| umask | Directory | File | Use Case |
|-------|-----------|------|----------|
| `022` | `755` (rwxr-xr-x) | `644` (rw-r--r--) | **Default** — Owner has full, others read-only |
| `002` | `775` (rwxrwxr-x) | `664` (rw-rw-r--) | **Team** — Group can write, others read-only |
| `027` | `750` (rwxr-x---) | `640` (rw-r-----) | **Restricted** — Only owner/group access |
| `077` | `700` (rwx------) | `600` (rw-------) | **Private** — Only owner can access |
| `000` | `777` (rwxrwxrwx) | `666` (rw-rw-rw-) | **Insecure** — Everyone has full access |


**📐 Base Permissions (If there is NO Umask / `000`)**


| Item Type | Base Value (No umask / `000`) | Resulting Permissions | What Users Can Do |
| :--- | :--- | :--- | :--- |
| **Directories** | `777` | `rwxrwxrwx` | Anyone can view, create, edit, and delete items inside. |
| **Files** | `666` | `rw-rw-rw-` | Anyone can read and edit the file, but nobody can run it. |


**umask vs chmod**

| Command | Scope | Timing | Purpose |
| :--- | :--- | :--- | :--- |
| **`umask`** | Default permissions for **new** files/folders | Applied **before** creation | **Preventive** measure |
| **`chmod`** | Modifies permissions on **existing** files/folders | Applied **after** creation | **Corrective** measure |


```bash
# View current umask value
umask
# Output: 0022

# Set umask to 022 (most common for admins)
umask 022

# Set umask to 002 (for team collaboration)
umask 002

# Check permissions after creating files with different umask
umask 022
touch file.txt
mkdir dir
ls -l
# file.txt: -rw-r--r-- (644)
# dir/: drwxr-xr-x (755)

umask 002
touch file2.txt
mkdir dir2
ls -l
# file2.txt: -rw-rw-r-- (664)
# dir2/: drwxrwxr-x (775)
```

## 2.3.8 ACLs (Access Control Lists) with `getfacl`/`setfacl`

In the traditional Unix permission model (User, Group, Others), if a specific user — who is neither the owner nor a member of the file's group — needs access to a file, you have only two choices: either add them to the group (which grants access to every group member) or change the file's ownership (which revokes access from the original owner). Both options are security risks because they over‑grant permissions.

ACLs solve this by letting you grant specific permissions (e.g., read, write, execute) to a single user without affecting anyone else — effectively giving that user "owner‑like" access without changing the actual ownership.



**Key Commands**

| Command | Purpose |
|---------|---------|
| `getfacl file` | View current ACLs on a file/directory |
| `setfacl -m rule file` | **Modify** — Add or change ACL entries |
| `setfacl -x rule file` | **Remove** — Delete specific ACL entries |
| `setfacl -b file` | **Remove all** — Clear all ACL entries |
| `setfacl -R` | **Recursive** — Apply to all subdirectories/files |


**ACL Syntax Rules**
```bash
# For users
setfacl -m u:username:permissions file

# For groups
setfacl -m g:groupname:permissions file
```

**Permissions format:**
- `rwx` — read, write, execute
- `rw-` — read, write
- `r--` — read-only
- `---` — no access

**Examples in Terminal**
```bash
-rw-rw-r--+ 1 root root 4096 Jun 27 15:57 data.json
          ^
     (Means an ACL is active on this file)
```
```bash
# View ACLs on a file
getfacl project.txt
# Output:
# # file: project.txt
# # owner: debayan
# # group: developers
# user::rw-
# group::r--
# other::r--

# Grant read/write access to a specific user
setfacl -m u:alice:rw project.txt

# Grant read-only access to a specific group
setfacl -m g:designers:r-- project.txt

# Grant execute permission to a specific user
setfacl -m u:john:rwx script.sh

# Remove ACL for a specific user
setfacl -x u:alice project.txt

# Remove all ACLs (back to standard permissions)
setfacl -b project.txt

# Recursively apply ACLs to a directory and all contents
setfacl -R -m u:alice:rwx /shared/project/

# Set default ACL (new files inherit this permission)
setfacl -m d:u:alice:rwx /shared/project/
```
